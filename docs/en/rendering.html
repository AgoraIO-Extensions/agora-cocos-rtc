Video Rendering

The only public render backend today is engine-texture: it uploads video frames as Cocos engine textures so you can composite video into your scene like any other texture.

The engine-texture backend

Call setRenderBackend('engine-texture') before initialize(...) to fix the render path.

engine-texture uploads video frames as Cocos engine textures, the render path whose output Cocos materials can actually consume. Reach for it when you need to crop, style, or deeply integrate video with your game UI; the cost is that your app handles the texture-ready events and binding logic described below.

displayNode and texture-ready events

With engine-texture, frames are uploaded into engine texture slots. When you pass displayNode into the video canvas, the SDK binds those textures to that node; otherwise you bind them to a Texture2D / SpriteFrame yourself via getEngineTexture(slotId). If the view stays blank, first check whether localVideoTextureReady and remoteVideoTextureReady fired as expected.

One caveat: when the event fires, the slot may still be initializing, so getEngineTexture(slotId) can briefly return null. Bind with a readiness check or a bounded retry instead of assuming the texture is immediately available.

Mirror and position semantics

Local-preview mirror behavior, observer position, and Android's limits around position semantics do not map one-to-one onto native SDK intuition. Mirroring is driven by mirrorMode on the video canvas (passed to setupLocalVideoView / setupRemoteVideoView) and on setVideoEncoderConfiguration, not by a dedicated method. When mirror or orientation is involved, rely on this doc set's described behavior rather than native SDK assumptions.

Compatibility surface

setNativeVideoOverlaySuspended(...) is a compatibility surface kept for native-overlay rendering. Under the public engine-texture path there is no native layer to hide, so calling it maps to no visible switch and your app does not need to rely on it.
