Rendering

The current render path is centered on engine-texture: video frames are uploaded as textures that Cocos can render.

Current backend support

The public render backend is engine-texture, so both integration and debugging should treat it as the default path.

displayNode and texture-ready events

When you pass displayNode into the video canvas, the SDK binds engine textures to that node. If the view stays blank, check whether localVideoTextureReady and remoteVideoTextureReady fired as expected.

Mirror and position semantics

This section covers mirror behavior, observer position, and Android limits around position semantics so you do not assume one-to-one native SDK behavior.

Compatibility surface

setNativeVideoOverlaySuspended still matters as part of the public surface, but in the current texture-driven model it is a compatibility API, not a visible native overlay switch.
